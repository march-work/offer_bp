# 应届生 Offer 评测器 — 算法完整拆解

> 本文档是 `development-report.md` 第三部分的详细展开，独立成册，供开发者实现时参照。

---

## 1. 设计动机

### 1.1 核心问题

应届生没有工作经验，面对一个 Offer 时缺乏参照系——不知道这个数字在市场上处于什么水平。

传统的工作价值评估公式 `Score = 收益 / 付出` 中，分母包含教育系数和经验倍数。但应届生经验=0 时分母偏小，导致评分虚高，不适合直接使用。

### 1.2 解法思路

**把"参照系"从常数变成动态计算值。**

- 传统公式：分子是你的收益，分母是常数 35 × 工时 × 教育 × 经验 → 绝对分值
- 应届生评测器：分子是你的收益，分母是**你应该拿到的** → 相对倍数

Score = 1.0 意味着你刚好拿到"你的背景应该拿到的水平"。

---

## 2. 最终确定公式

```
FreshGrad Score = (dailySalary × envFactor) / (expectedDailySalary × timeFactor)
```

其中 `timeFactor = effectiveHours / 8`，归一化到标准 8 小时工作制。

### 2.1 公式结构拆解

```
┌─────────────────────────── 分子 ───────────────────────────┐
│  dailySalary × envFactor                                  │
│  你实际获得的（日薪 × 环境加分）                              │
└────────────────────────────────────────────────────────────┘
                            ÷
┌─────────────────────────── 分母 ───────────────────────────┐
│  expectedDailySalary × (effectiveHours / 8)                │
│  你应该获得的（期望日薪 × 工时倍率）                           │
└────────────────────────────────────────────────────────────┘
```

### 2.2 各项含义

| 变量 | 含义 | 计算来源 |
|------|------|----------|
| `dailySalary` | 实际标准化日薪 | `年薪 / 实际年工作日数` |
| `envFactor` | 工作环境综合系数 | 6 个子因子相乘 |
| `expectedDailySalary` | 市场期望日薪 | `期望年薪 / 260`（260 = 52周×5天） |
| `effectiveHours` | 每日有效工时 | `workHours + 通勤 - 0.5 × 休息` |
| `timeFactor` | 工时倍率 | `effectiveHours / 8`，标准 8h = 1.0 |

### 2.3 为什么用 effectiveHours / 8？

直观理解：
- `timeFactor = 1.0`（每天 8h）→ 标准工作制，无工时惩罚
- `timeFactor = 1.5`（每天 12h）→ 分母变大 1.5 倍，Score 下降 → 工时长 = 惩罚
- `timeFactor = 0.875`（每天 7h）→ 分母变小，Score 上升 → 工时短 = 加分

Score > 1 = 你每付出一个标准工时，获得的报酬超过了你的背景应该拿的水平。

## 3. 子公式详解

### 3.1 dailySalary — 实际日薪

```
dailySalary = 年薪 / 实际年工作日数
```

其中年薪 = 月薪 × 月数 + 年终奖 + 股票(×万) + 补贴 × 12。

#### 3.1.1 实际年工作日数

```
实际年工作日 = 52 × 每周工作天数 - (年假 + 法定假日 + 带薪病假 × 0.6)
```

**参数说明**：

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `每周工作天数` | 选择 | 5 | 4/5/6/7 |
| `年假` | 数字 | 5 | 天 |
| `法定假日` | 数字 | 13 | 天（中国标准） |
| `带薪病假` | 数字 | 3 | 乘以 0.6 因为并非 100% 利用 |

**计算示例**：
```
52 × 5 - (5 + 13 + 3 × 0.6) = 260 - 19.8 = 240.2 天
```

#### 3.1.2 完整示例

```
年薪 = ¥220,000, 年工作日 = 240.2
dailySalary = 220,000 / 240.2 = ¥915.90
```
---

### 3.2 envFactor — 环境系数

```
envFactor = 工作环境 × 领导关系 × 同事关系 × 城市储蓄系数 × 食堂系数 × 定居系数 × 地点偏好
```

7 个子因子均为乘法关系。

#### 3.2.1 子因子取值表

**工作环境**：

| 选项 | 系数 | 说明 |
|------|------|------|
| 高端园区 | 1.2 | Google 级别 |
| CBD / 甲级写字楼 | 1.1 | |
| 普通 | 1.0 | 基准 |
| 偏远 / 厂区 | 0.9 | |
| 条件较差 | 0.8 | |

**领导关系**：

| 选项 | 系数 |
|------|------|
| 善解人意 | 1.2 |
| 比较好 | 1.1 |
| 中规中矩 | 1.0 |
| 比较坑 | 0.9 |
| 简直噩梦 | 0.8 |

**同事关系**：

| 选项 | 系数 |
|------|------|
| 亲如一家 | 1.2 |
| 和和睦睦 | 1.1 |
| 萍水相逢 | 1.0 |
| 勾心斗角 | 0.9 |
| 乌烟瘴气 | 0.8 |

**城市储蓄系数**：基于城市真实收入/消费/房价数据动态计算，不再使用城市分级硬编码。

```
储蓄率 = (城市人均收入 - 消费 × 0.7 - 年居住成本) / 城市人均收入
归一化值 = 储蓄率 / 全国平均储蓄率
城市储蓄系数 = 分段线性映射 [0.6, 1.2]
```

**定居系数**：

```
定居系数 = 10年收入 / 新房90㎡首付(30%)
系数 >= 3 → 1.2（定居轻松，加分）
系数 <= 1 → 0.8（定居困难，减分）
1 < 系数 < 3 → 线性插值 0.8 ~ 1.2
```

**食堂系数**：

| 选项 | 系数 |
|------|------|
| 丰富且便宜 | 1.15 |
| 不错 | 1.1 |
| 普通 | 1.0 |
| 较差 | 0.9 |
| 无食堂 | 1.0 |

**地点偏好**：

| 选项 | 系数 |
|------|------|
| 喜欢/离家近 | 1.1 |
| 无所谓 | 1.0 |
| 不太满意/有点远 | 0.9 |

#### 3.2.2 完整示例

```
工作环境 = CBD (1.1)
领导 = 善解人意 (1.2)
同事 = 和睦 (1.1)
城市储蓄系数 = 0.9（成都，基于真实数据计算）
食堂 = 无 (1.0)
定居系数 = 1.1
地点偏好 = 无所谓 (1.0)

envFactor = 1.1 × 1.2 × 1.1 × 0.9 × 1.0 × 1.1 × 1.0 = 1.438
```

### 3.3 expectedDailySalary — 期望日薪（核心创新）

```
期望年薪 = 基础年薪 × 行业系数
期望日薪 = 期望年薪 / 260
```

这是应届生评测器的核心创新：用一个动态计算的"你应该拿多少"替代传统的固定常数基准。

> **注意**：城市差异已通过行业薪资的真实城市数据隐式体现（如北京的"信息传输"行业均薪高于成都），不再使用城市分级硬编码系数。

#### 3.3.1 基础年薪表

基于应届生市场薪资数据（2024-2025 中国市场参考）：

| 学历 | 学校等级 | 基础年薪（万元） |
|------|----------|-----------------|
| 专科及以下 | — | 6 |
| 本科 | 二本三本 | 8 |
| 本科 | 双非一本 | 10 |
| 本科 | 985/211 | 15 |
| 本科 | C9/清北 | 25 |
| 硕士 | 二本本科 + 二本硕士 | 10 |
| 硕士 | 双非本科 + 双非硕士 | 12 |
| 硕士 | 211本科 + 211硕士 | 18 |
| 硕士 | 985本科 + 985硕士 | 25 |
| 硕士 | C9/清北 | 35 |
| 博士 | 普通 | 18 |
| 博士 | 985/QS50 | 30 |
| 博士 | C9/清北 | 45 |

> **数据来源**：各大高校就业质量报告、Boss直聘/拉勾校招薪资数据、脉脉统计。建议预留数据更新机制。

> **设计决策**：硕士区分本科和硕士的学校层次（组合逻辑），博士只看最高学历学校。

#### 3.3.2 行业系数

行业系数 = 城市该行业平均薪资 / 城市全行业平均薪资（使用真实数据）。当无真实数据时回退到硬编码兜底值。

**硬编码兜底值**（仅用于无真实城市数据时）：

| 行业 | 系数 | 说明 |
|------|------|------|
| 信息传输、软件和信息技术服务 | 1.20 | 传统高薪 |
| 科学研究和技术服务 | 1.15 | 技术密集 |
| 金融 | 1.10 | 稳定高薪 |
| 卫生和社会工作 | 1.05 | 稳健增长 |
| 文化、体育和娱乐 | 0.95 | 一般 |
| 制造 | 0.90 | 传统行业 |
| 教育 | 0.80 | 受政策影响 |
| 其他 | 0.85 | 默认 |

#### 3.3.3 完整示例

```
学历 = 本科 985 → 基础年薪 = 15万
Offer城市 = 成都
你的专业 = 信息传输、软件和信息技术服务 → 成都该行业均薪/全行业均薪 = 1.18（真实数据）

期望年薪 = 15 × 1.18 = 17.7万
期望日薪 = 177,000 / 260 = ¥680.77
```

### 3.4 effectiveHours — 有效工时

```
effectiveHours = workHours + commute × officeRatio × shuttleFactor - 0.5 × restTime
```

#### 3.4.1 各项参数

**workHours** — 日均工时（小时），用户直接输入。

**commute** — 通勤时间（小时），用户直接输入。

**officeRatio** — 办公室比例：
```
officeRatio = (每周工作天数 - WFH天数) / 每周工作天数
```
- WFH 天不需要通勤，按比例折算。
- 示例：每周 5 天，WFH 2 天 → `officeRatio = (5-2)/5 = 0.6`

**shuttleFactor** — 班车系数：

| 有无班车 | 系数 |
|----------|------|
| 无 | 1.0 |
| 有班车 | 0.3（通勤时间变为 30%） |

**restTime** — 休息时间（小时），每日工作中的午休/休息。

**0.5 × restTime** — 休息时间只有一半算作"付出"，因为这部分并非高强度劳动。

#### 3.4.2 完整示例

```
workHours = 9h
commute = 1h
每周 5 天, WFH 0 天 → officeRatio = (5-0)/5 = 1.0
无班车 → shuttleFactor = 1.0
restTime = 1.5h

effectiveHours = 9 + 1 × 1.0 × 1.0 - 0.5 × 1.5 = 9.25h
```

## 4. 评级体系

| 分值范围 | 评级 | 颜色（Tailwind） | 含义 |
|----------|------|------------------|------|
| < 0.5 | 大冤种 | `text-pink-800` | 严重低于市场，快跑 |
| 0.5 ~ 0.7 | 偏低 | `text-red-500` | 明显低于期望 |
| 0.7 ~ 0.9 | 一般 | `text-orange-500` | 略低于期望，可以谈 |
| 0.9 ~ 1.1 | 合理 | `text-blue-500` | 符合市场水平 |
| 1.1 ~ 1.3 | 不错 | `text-green-500` | 超出期望 |
| 1.3 ~ 1.6 | 很香 | `text-purple-500` | 远超期望 |
| ≥ 1.6 | 天选 Offer | `text-yellow-400` | 顶级 offer |

**评级边界说明**：
- Score = 1.0 是核心锚点，对应"刚好等于市场期望"
- 0.9~1.1 是"合理区间"，接受不会有明显亏欠感
- 向下区间（<0.9）更密集，因为"亏了"的感知更敏感
- 向上区间（>1.1）跨度更大，高薪 offer 稀缺

## 6. 完整计算示例（逐步演示）

### 场景：成都 985 本科，互联网公司 Offer

#### 输入参数

```
学历: 本科, 学校等级: 985/211
Offer城市: 成都, 你的专业: 信息传输、软件和信息技术服务
年薪: ¥220,000
每周工作: 5天, WFH: 0天
年假: 5天, 法定假日: 13天, 带薪病假: 3天
日均工时: 9h, 通勤: 1h, 休息: 1.5h
工作环境: CBD (1.1), 领导: 善解人意 (1.2), 同事: 和睦 (1.1)
城市储蓄系数: 0.9, 定居系数: 1.1, 地点偏好: 无所谓 (1.0)
无班车、无食堂
```

#### 第 ① 步：计算实际年工作日

```
年工作日 = 52 × 5 - (5 + 13 + 3 × 0.6)
         = 260 - 19.8
         = 240.2 天
```

#### 第 ② 步：计算实际日薪

```
dailySalary = 220,000 / 240.2 = ¥915.90
```

#### 第 ③ 步：计算环境系数

```
envFactor = 1.1 × 1.2 × 1.1 × 0.9 × 1.0 × 1.1 × 1.0 = 1.438
```

#### 第 ④ 步：计算期望日薪

```
期望年薪 = 15万 × 1.18(成都IT行业因子，真实数据) = 17.7万
期望日薪 = 177,000 / 260 = ¥680.77
```

#### 第 ⑤ 步：计算有效工时

```
officeRatio = (5 - 0) / 5 = 1.0
effectiveHours = 9 + 1 × 1.0 × 1.0 - 0.5 × 1.5
               = 9 + 1 - 0.75
               = 9.25h
timeFactor = 9.25 / 8 = 1.156
```

#### 第 ⑥ 步：计算最终得分

```
Score = (dailySalary × envFactor) / (expectedDailySalary × timeFactor)
      = (915.90 × 1.438) / (680.77 × 1.156)
      = 1,317.06 / 787.37
      ≈ 1.67 → 评级「天选 Offer」
```

#### 第 ⑦ 步：对比不同工时

如果同样的 ¥22万 offer，但 996 工作制：

```
effectiveHours = 12 + 1 - 0.5 × 1 = 12.5h
timeFactor = 12.5 / 8 = 1.5625
Score = (915.90 × 1.438) / (680.77 × 1.5625)
      = 1,317.06 / 1,063.70
      ≈ 1.24 → 评级「不错」
```

**结论**：同样是 ¥22万 offer，955 是"天选"(1.67)，996 是"不错"(1.24)。工时惩罚自然生效。

## 7. 公式推导历史（设计决策记录）

开发过程中经历了多次公式修正，记录在此以供理解设计意图。

### 7.1 第一版（已废弃）

```
Score = (dailySalary × envFactor) / (expectedDailySalary × effectiveHours × baseEnvFactor)
```

**问题**：Score ≈ 0.15，太低。因为期望日薪是"日薪"级别，再乘以 effectiveHours（约 9~12），分母过大。

### 7.2 第二版（已废弃）

```
Score = (dailySalary × envFactor) / (expectedDailySalary × baseEnvFactor)
```

**问题**：工时因素被完全忽略。同样是 ¥15万，955 和 996 算出相同分数。

### 7.3 第三版（已废弃）

```
Score = (dailySalary × envFactor × 8) / (expectedDailySalary × effectiveHours)
```

**问题**：数学等价但不直观，"× 8"的物理含义需要额外解释。

### 7.4 最终版（当前采用）

```
Score = (dailySalary × envFactor) / (expectedDailySalary × timeFactor)
```

其中 `timeFactor = effectiveHours / 8`。

**解决方案**：将 effectiveHours 除以 8 归一化为"工时倍率"，直观含义清晰：
- timeFactor = 1.0 → 标准 8h，无惩罚
- timeFactor > 1.0 → 加班，分母变大，Score 下降
- timeFactor < 1.0 → 工时短，分母变小，Score 上升

推导逻辑：

```
分子 = dailySalary × envFactor
     = (年薪/年工作日) × 环境
     = "你的日收益（含环境加分）"

分母 = expectedDailySalary × timeFactor
     = (期望年薪/260) × (有效工时/8)
     = "你背景对应的日薪 × 工时倍率"
```

当有效工时 > 8 时，timeFactor > 1 → 分母变大 → Score 下降 → 工时长 = 惩罚。

| 经验倍数 | 应届生=0，不需要 |

## 9. 边界情况与注意事项

### 9.1 边界情况

| 场景 | 处理方式 |
|------|----------|
| 学历=专科 | 基础年薪=6万，Score 可能偏高（低期望易达到） |
| 城市=其他 + 行业=教育 | 期望薪资极低（无真实数据时回退兜底值），Score 可能虚高 |
| 年薪=0 | 提示用户必须输入有效薪资 |
| WFH=全部天数 | 通勤=0，officeRatio=0，无通勤成本 |
| 食堂=丰富且便宜 + 班车=有 | envFactor 会显著偏高 |

### 9.2 与跳槽评测器 Score 的不可直接比较

应届生评测器的 Score 1.0 ≠ 跳槽评测器的 Score 1.0。两者公式结构不同，分母基准不同。不要在 UI 上直接并列对比两个 Score。

### 9.3 数据更新

期望薪资的基础年薪表具有时效性（校招薪资每年变化）。建议：
- 硬编码当前版本数据（2024-2025）
- 预留数据更新接口
- 每年 Q1 更新一次

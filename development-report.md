## 三、评测器一：应届生 Offer 评测器

### 3.1 设计理念

应届生的核心困境是**没有参照系**——不知道这个 offer 在市场上处于什么水平。

Worth Calculator 的公式 `Score = 收益 / 付出` 中，"付出"部分的分母包含教育系数和经验倍数，这对应届生来说其实不太合理（教育系数=1.0 + 经验=0 时分母偏小，会导致应届生评分虚高）。

应届生评测器的改进思路：
1. **分母改为"期望薪资"**：用个人背景（学历+学校+城市+行业）计算出一个合理的市场期望基准，作为分母
2. **分子仍然是"收益"**：标准化日薪 × 环境系数，与 Worth Calculator 一致
3. **最终得分含义**：你的 Offer 超过/低于市场期望的倍数

### 3.2 核心公式

```
FreshGrad Score = (标准化日薪 × 环境系数) / (期望日薪 × 有效日工时 × 环境基线)
```

展开：

```
Score = (dailySalary × envFactor) / (expectedDailySalary × effectiveHours × baseEnvFactor)
```

- **dailySalary**: 与 Worth Calculator 完全一致的 PPP 标准化日薪
- **envFactor**: 与 Worth Calculator 完全一致的环境系数
- **expectedDailySalary**: 基于个人背景计算的市场期望日薪（替代 Worth Calculator 中的常数 35）
- **effectiveHours**: 与 Worth Calculator 完全一致的有效日工时
- **baseEnvFactor**: 环境基线（=1.0），用于归一化

**得分的含义**：

| 得分范围 | 含义 |
|----------|------|
| Score > 1.2 | Offer 薪资超出市场期望 20%+，非常好 |
| Score = 1.0 | Offer 刚好等于市场期望，合理 |
| Score = 0.8 | Offer 低于市场期望 20%，偏低 |
| Score < 0.6 | Offer 远低于市场期望，很坑 |

### 3.3 期望日薪计算（核心创新）

**函数**: `calculateExpectedSalary()`

```
期望年薪 = 基础年薪 × 城市系数 × 行业系数
期望日薪 = 期望年薪 / 年工作日数（260天基准）
```

### 3.4 公式完整拆解

```
FreshGrad Score = (dailySalary × envFactor) / (expectedDailySalary × effectiveHours)

其中:
  dailySalary = (年薪 × PPP_factor) / 年工作日数
  envFactor = 工作环境 × 领导关系 × 同事关系 × 城市系数 × 食堂系数
  expectedDailySalary = 基础年薪 × 城市系数 × 行业系数 / 260
  effectiveHours = workHours + commuteHours × officeRatio × shuttleFactor - 0.5 × restTime
```

**与 Worth Calculator 公式的对比**：

| 组成部分 | Worth Calculator | 应届生评测器 |
|----------|-----------------|-------------|
| 分子 | dailySalary × envFactor | dailySalary × envFactor（**完全一致**） |
| 分母 - 常数 | 35 | expectedDailySalary（**动态基准**） |
| 分母 - 工时 | effectiveHours | effectiveHours（**完全一致**） |
| 分母 - 教育 | education factor | **已融入期望薪资** |
| 分母 - 经验 | experience multiplier | **不需要（应届生=0）** |

**关键变化**: Worth Calculator 中的"教育系数"和"经验倍数"被替换为"期望日薪"。原因是：
- Worth Calculator 的教育系数在分母中，高学历者同薪资下分数更低——这对已从业者合理
- 应届生评测器直接把教育背景转化为一个"你应该拿多少"的基准，更直观

### 3.7 个性化报告维度

`FreshGradShareCard` 生成以下 7 大维度评价：

1. **最终评估** — 总分 + 评级 + 与市场期望的差距百分比
2. **薪资评价** — 你的年薪 vs 期望年薪，你在同届中大约处于什么分位
3. **城市选择** — 城市等级与薪资的匹配度
4. **行业选择** — 行业前景与薪资水平分析
5. **工作体验** — 工时 + 通勤 + WFH + 环境 + 人际关系的综合评价
6. **性价比总结** — 综合薪资和工作体验的整体性价比
7. **行动建议** — 接受 / 谈判 / 继续寻找 的具体建议

### 3.8 算法示例

#### 示例：成都 985 本科拿互联网 Offer

**输入参数**:
- 学历: 本科 985
- 目标城市: 成都（新一线）
- 目标行业: 互联网
- 年薪: ¥220,000
- 国家: 中国 (PPP=4.19)
- 每周工作: 5天, WFH: 0天
- 年假: 5天, 法定假日: 13天, 带薪病假: 3天
- 日均工时: 9h, 通勤: 1h, 休息: 1.5h
- 工作环境: CBD (1.1), 城市: 新一线 (0.80)
- 领导: 善解人意 (1.1), 同事: 和和睦睦 (1.1)
- 无班车、无食堂

**计算过程**:

```
① 年工作日 = 52 × 5 − (5 + 13 + 3 × 0.6) = 260 − 19.8 = 240.2 天

② 标准化日薪 = 220,000 / 240.2 = ¥915.90

③ 环境系数 = 1.1 × 1.1 × 1.1 × 0.80 × 1.0 = 1.0648

④ 期望年薪 = 15万 × 1.0(新一线) × 1.2(互联网) = 18万
   期望日薪 = 180,000 / 260 = ¥692.31

⑤ 有效工时 = 9 + 1×1.0×1.0 − 0.5×1.5 = 9.25h

⑥ Score = (915.90 × 1.0648) / (692.31 × 9.25)
         = 975.22 / 6403.87
         = 0.1523

...等等，这个得分太低了。问题在哪？
```

**发现问题**: 期望日薪不应该再乘以工时。Worth Calculator 中常数 35 本身已经是一个"日薪基准"，不需要额外乘工时。让我修正公式。

### 3.9 公式修正

回顾 Worth Calculator 的公式：
```
Score = (dailySalary × envFactor) / (35 × effectiveHours × education × experience)
```

这里 `35 × effectiveHours` 的含义是：35 是"理想时薪基准"，乘以有效工时得到"理想日薪"。但 Worth Calculator 的分子是 dailySalary（日薪），所以实际上是在比较：

```
实际日薪 / 理想日薪 = 实际时薪 / 理想时薪
```

对于应届生评测器，我需要同样的结构：

```
Score = (dailySalary × envFactor) / (expectedHourlyWage × effectiveHours)
```

但更简洁的做法是——**直接比较日薪**：

```
Score = (dailySalary × envFactor) / (expectedDailySalary × envBaseFactor)
```

这里 envBaseFactor 用来归一化环境的影响，取 1.0。这样当环境一般（envFactor=1.0）时，Score = dailySalary / expectedDailySalary，非常直观。

**但这意味着工时因素被完全忽略了！** 这不合理——同样是 ¥15万 的 offer，955 和 996 的价值完全不同。

**最终方案**: 保留工时，但期望日薪也包含工时因素：

```
Score = (dailySalary × envFactor) / (expectedDailySalary × envBaseFactor)
```

其中期望日薪的计算已经隐含了标准工时假设（8h/天，5天/周），分子中的日薪是按实际工时算的。所以：
- 如果实际工时 = 8h（与期望一致），Score 主要反映薪资水平
- 如果实际工时 > 8h，同样的年薪 → 日薪更低（年工作日可能也更少），Score 自然下降

**验证**: 
```
年工作日 = 240 天（5天×52周 - 20天假期）
标准年工作日 = 260 天（5天×52周 - 0天假期，基准假设）

实际日薪 = 220,000 / 240 = ¥916.67
期望日薪 = 180,000 / 260 = ¥692.31

环境系数 = 1.1 × 1.1 × 1.1 × 0.80 = 1.0648

Score = (916.67 × 1.0648) / (692.31 × 1.0)
       = 975.76 / 692.31
       ≈ 1.41 → 评级「很香」🟣
```

**这更合理了！** Offer 年薪 ¥22万（期望 ¥18万），超出了 22%，又因为环境不错（CBD + 好领导同事），最终 1.41 分。

#### 修正后的公式

```
FreshGrad Score = (dailySalary × envFactor) / (expectedDailySalary × baseEnvFactor)

其中:
  dailySalary = 年薪 / 实际年工作日数
  expectedDailySalary = 期望年薪 / 标准年工作日数（260天）
  envFactor = 工作环境 × 领导关系 × 同事关系 × 城市系数 × 食堂系数
  baseEnvFactor = 1.0（归一化常数）
```

> **注意**: 这里的公式比 Worth Calculator 简单——去掉了 effectiveHours 的显式计算。原因是**工时和假期的效果已经通过 dailySalary 隐含了**（工时多 → 年工作日少 → 日薪高 → 但这个"高"是假的，因为你是用更多劳动换来的）。不过等一下，这不对——如果年薪固定，工时长并不会减少年工作日数...

**再次修正**: 工时长确实不影响年工作日数（年工作日 = 52×每周天数 - 假期，与日均工时无关）。所以两个 22万年薪的 offer，一个 955 一个 996，算出来的 dailySalary 完全一样。

**最终最终方案**: 必须显式引入工时因素。回归 Worth Calculator 的框架：

```
FreshGrad Score = (dailySalary × envFactor) / (expectedDailySalary × effectiveHours / standardHours × baseEnvFactor)
```

其中 `standardHours = 8`（标准工时）。

简化为：

```
FreshGrad Score = (dailySalary × envFactor) / (expectedDailySalary × (effectiveHours / 8) × baseEnvFactor)
```

或者更清晰地写：

```
FreshGrad Score = (dailySalary × envFactor × 8) / (expectedDailySalary × effectiveHours × baseEnvFactor)
```

**最终验证**:
```
dailySalary = 220,000 / 240.2 = ¥915.90
envFactor = 1.0648
effectiveHours = 9 + 1 - 0.5×1.5 = 9.25h
expectedDailySalary = 180,000 / 260 = ¥692.31

Score = (915.90 × 1.0648 × 8) / (692.31 × 9.25 × 1.0)
      = 7,801.88 / 6,403.87
      ≈ 1.218 → 评级「不错」🟢
```

如果工时变成 12h（996）:
```
effectiveHours = 12 + 1 - 0.5×1 = 12.5h
Score = (915.90 × 1.0648 × 8) / (692.31 × 12.5)
      = 7,801.88 / 8,653.88
      ≈ 0.902 → 评级「一般」🟠
```

**完美！** 同样的 ¥22万 offer，955 是"不错"，996 是"一般"。工时惩罚自然生效。

### 3.10 最终公式（确定版）

```
FreshGrad Score = (dailySalary × envFactor × 8) / (expectedDailySalary × effectiveHours)

dailySalary = (年薪 × PPP标准化) / 实际年工作日数
expectedDailySalary = 基础年薪 × 城市系数 × 行业系数 / 260
envFactor = 工作环境 × 领导关系 × 同事关系 × 城市系数 × 食堂系数
effectiveHours = workHours + commute × officeRatio × shuttleFactor - 0.5 × restTime
```

**公式结构总结**:
```
分子 = 你实际获得的（日薪 × 环境加分 × 标准工时8h）
分母 = 你应该获得的（期望日薪 × 实际付出的工时）
```

Score = 1.0 意味着你刚好拿到"你的背景应该拿到的水平"。

## 四、评测器二：跳槽决策评测器

### 4.1 设计理念

跳槽者的核心问题是**增量到底有多大**。新 offer 年薪多了 5 万，但公积金少了、通勤远了、试用期有风险——到底划不划算？

设计思路：
1. **对当前工作和新 Offer 分别计算 Worth Score**（复用 Worth Calculator 的公式）
2. **做差得到增量**：`Δ = Score_new - Score_current`
3. **扣除转换成本**：`Δ_net = Δ - Transition_Cost_Score`
4. **根据净增量给出建议**

### 4.2 核心公式

```
跳槽决策分 = Score_new - Score_current - 转换成本惩罚

其中:
  Score_new = Worth Calculator 公式套用在新 Offer 上
  Score_current = Worth Calculator 公式套用在当前工作上
  转换成本惩罚 = 综合转换成本的标准化惩罚分
```

#### 4.2.1 单岗评分（复用 Worth Calculator）

对于当前工作和新 Offer，各自独立计算 Worth Score：

```
Score(job) = (dailySalary × envFactor) / (35 × effectiveHours × education × experience)
```

完全复用 Worth Calculator 的公式，包括：
- PPP 标准化
- 年工作日计算
- 环境系数（工作环境、领导、同事、城市、食堂、班车）
- 教育系数
- 经验薪资倍数

#### 4.2.2 转换成本惩罚

转换成本是跳槽评测器的核心创新。将其标准化为一个 0~2.0 的惩罚分：

```
转换成本惩罚 = 财务成本惩罚 + 风险惩罚

财务成本惩罚:
  = (未兑现奖金 + 未归属股票 + 空窗期损失 + 福利差额) / 当前年薪 × 2
  // 标准化为 0~1 的比例，乘以 2 放大影响

风险惩罚:
  = P(试用期不通过) × 0.5
  // 最高 0.5 分的惩罚
```

**各项详细计算**:

| 成本项 | 计算方式 | 示例 |
|--------|----------|------|
| **未兑现奖金** | 未发奖金金额 | ¥30,000 年终奖已工作8个月，未发 = ¥30,000 |
| **未归属股票** | 未 vest 股票估值 | ¥50,000 RSU 分4年 vest，剩 ¥37,500 |
| **空窗期损失** | 空窗天数/365 × 当前年薪 | 2周空窗 = 14/365 × ¥300,000 = ¥11,507 |
| **福利年差额** | (公积金差 + 补贴差) × 12 | 公积金从12%→5%，基数¥20,000 = (0.12-0.05)×20000×2×12 = ¥33,600 |
| **试用期风险** | P_fail × 0.5 | 大厂 P=0.05 → 0.025 |

**财务成本惩罚示例**:
```
总财务成本 = 30,000 + 37,500 + 11,507 + 33,600 = ¥112,607
当前年薪 = ¥300,000
财务成本惩罚 = 112,607 / 300,000 × 2 = 0.751

风险惩罚 = 0.025（大厂低风险）

转换成本惩罚 = 0.751 + 0.025 = 0.776
```

#### 4.2.3 转换成本简化输入

用户不需要手动计算所有子项。采用智能分层：

**必填**（4 项）:
1. 未发放的年终奖/绩效奖金（金额）
2. 未归属的股票/期权（金额，0=无）
3. 预计空窗期（0天 / 1周 / 2周 / 1月 / 2月 / 3月+）
4. 试用期风险（低 P=0.05 / 中 P=0.15 / 高 P=0.30）

**选填**（2 项）:
5. 新旧公司公积金比例差异（0%=一样 / 12%→5% 等）
6. 年度补贴差异（金额，正数=新公司多，负数=新公司少）

**自动计算**（无需输入）:
- 福利差额自动从公积金比例差异算出

### 4.6 算法示例

#### 示例：北京互联网人跳槽

**当前工作**:
- 年薪: ¥300,000 | 城市: 北京（一线） | 私企 | 3-5年
- 工时: 10h | 通勤: 1.5h | 休息: 1.5h | WFH: 2天
- 环境: CBD (1.1) | 领导: 中规中矩 (1.0) | 同事: 萍水相逢 (1.0)
- 城市系数: 0.70 | 无班车、无食堂
- 学历: 本科双非 (1.0)

**新 Offer**:
- 年薪: ¥380,000 | 城市: 杭州（一线） | 私企 | 3-5年
- 工时: 9h | 通勤: 1h | 休息: 1h | WFH: 2天
- 环境: CBD (1.1) | 领导: 善解人意 (1.1) | 同事: 和睦 (1.1)
- 城市系数: 0.70 | 有食堂不错 (1.1) | 无班车

**转换成本**:
- 未发年终奖: ¥25,000
- 未归属股票: 0
- 空窗期: 2周 ≈ ¥11,507
- 试用期风险: 低 (0.05)
- 公积金差异: 12%→8%, 基数按¥25,000算 = (0.12-0.08)×25000×2×12 = ¥24,000/年
  （此为年差额，跳槽前只损失"已工作月份对应的差额"，此处简化为全额计入）

**计算过程**:

```
─── 当前工作 ───

年工作日 = 52×5 − (10+13+5×0.6) = 260 − 26 = 234 天
日薪 = 300,000 / 234 = ¥1,282.05

办公室比例 = (5−2)/5 = 0.6
有效通勤 = 1.5 × 0.6 × 1.0 = 0.9h
有效工时 = 10 + 0.9 − 0.5×1.5 = 10.15h

环境系数 = 1.1 × 1.0 × 1.0 × 0.70 × 1.0 = 0.77

教育系数 = 1.0（本科双非）
经验倍数 = 1 + (2.7−1) × 1.0 = 2.7

Score_current = (1282.05 × 0.77) / (35 × 10.15 × 1.0 × 2.7)
              = 987.18 / 959.18
              ≈ 1.029

─── 新 Offer ───

年工作日 = 52×5 − (10+13+5×0.6) = 260 − 26 = 234 天
日薪 = 380,000 / 234 = ¥1,623.93

办公室比例 = (5−2)/5 = 0.6
有效通勤 = 1.0 × 0.6 × 1.0 = 0.6h
有效工时 = 9 + 0.6 − 0.5×1 = 9.1h

环境系数 = 1.1 × 1.1 × 1.1 × 0.70 × 1.1 = 1.0287

Score_new = (1623.93 × 1.0287) / (35 × 9.1 × 1.0 × 2.7)
          = 1670.40 / 860.85
          ≈ 1.941

─── 转换成本 ───

总财务成本 = 25,000 + 0 + 11,507 + 24,000 = ¥60,507
财务成本惩罚 = 60,507 / 300,000 × 2 = 0.403
风险惩罚 = 0.05 × 0.5 = 0.025
转换成本惩罚 = 0.403 + 0.025 = 0.428

─── 最终决策 ───

跳槽决策分 = 1.941 − 1.029 − 0.428 = 0.484
→ 评级「值得跳」🟢
```

**分析**: 新 Offer 的 Worth Score (1.94) 远高于当前工作 (1.03)，即使扣除 ¥6万的转换成本，净增量仍为正。建议接受。

## 七、关键设计特点

### 7.1 期望薪资模型（应届生评测器核心创新）

- 基于学历 × 学校 × 城市 × 行业四维计算
- 数据来自公开的校招薪资报告
- 作为动态基准替代 Worth Calculator 的固定常数 35
- 让应届生有"我应该拿多少"的直观参考

### 7.2 转换成本量化（跳槽评测器核心创新）

- 将模糊的"跳槽成本"转化为具体数字
- 分层设计：4 项必填 + 2 项选填
- 财务成本标准化为惩罚分
- 试用期风险概率化

### 7.3 公式一致性

- 两个评测器的环境系数、工时计算、PPP 逻辑与 Worth Calculator 完全一致
- 用户在同一产品体系内，三个工具的评分可以互相参照
- 降低用户的学习成本

### 7.4 独立运行

- 两个评测器页面完全独立，各自有独立的 Calculator 和 ShareCard 组件
- 互不依赖，可以独立迭代
- 共享的代码通过 `lib/` 目录中的工具函数复用

| **主观字段偏差** | 环境评估依赖用户自评 | 提供"不确定选中间值"的默认行为 |
| **期权估值困难** | 早期公司期权价值不确定 | 转换成本中默认保守估值 |
| **跳槽公式依赖 Worth Calculator** | 如果 WC 公式有偏差，跳槽评测器也会偏差 | 独立验证跳槽场景的合理性 |

---

## 十、总结

Offer BP 包含两个独立评测器，各自遵循 Worth Calculator 的成熟模式（单一公式 + 评级体系 + 个性化报告），但在算法层面做了针对性创新：

**应届生评测器**:
1. 用**动态期望薪资**替代 Worth Calculator 的常数 35
2. 期望薪资 = 学历 × 学校 × 城市 × 行业，让应届生有直观参考
3. 公式保持 `分子/分母` 结构，含义变为"你拿到的 vs 你应该拿到的"

**跳槽评测器**:
1. 对当前工作和新 Offer **分别计算 Worth Score**
2. 做差得到增量，扣除**量化转换成本**
3. 转换成本包括未兑现奖金、未归属股票、空窗期损失、公积金差额、试用期风险
4. 最终输出"跳 / 留"的行动建议

**两个评测器的核心共同点**: 都复用 Worth Calculator 的环境系数、工时计算和 PPP 标准化逻辑，保证产品体系的一致性。
